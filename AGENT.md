# CampusRelay AI Agent Instruction (MVP → SaaS 확장용)

## 1. 프로젝트 목적

CampusRelay는 학교 단위 무료 나눔/거래 플랫폼이다.

핵심 목표:

* 학교 기반 사용자 매칭
* 안전한 거래 (학생 중심)
* 간단한 등록 / 빠른 탐색
* MVP → 일본 전국 확장

---

## 2. 기술 스택 (고정)

* Frontend: React (Vite)
* Backend: Supabase (DB + Auth + Storage)
* Hosting: Cloudflare Pages
* 지도: Google Maps API

👉 FastAPI는 필요 시에만 추가 (기본은 Supabase 중심)

---

## 3. 핵심 기능 (우선순위 기준)

### ① 사용자

* 이메일/소셜 로그인 (Supabase Auth)
* 학교 인증 (도메인 또는 수동 승인)
* 프로필 (학교, 학년 선택)

### ② 게시글

* 물품 등록 (사진 + 설명 + 가격)
* 무료 나눔 / 판매 구분
* 카테고리 (교복 / 책 / 기타)
* 학교별 필터

### ③ 거래

* 채팅 (기본 텍스트)
* 거래 상태 (대기 / 진행중 / 완료)

### ④ 지도

* 근처 거래 표시 (Google Maps)
* 학교 중심 필터

---

## 4. DB 설계 원칙 (Supabase)

필수 테이블:

* users
* schools
* posts
* chats
* transactions

규칙:

* 모든 데이터는 user_id 기반
* 학교 단위로 접근 제한 (RLS 필수)
* 삭제 대신 상태값 사용 (soft delete)

---

## 5. 보안 (중요)

* RLS(Row Level Security) 반드시 적용
* 동일 학교 사용자만 게시글 조회 가능
* 본인 데이터만 수정 가능
* 이미지 업로드는 사용자별 경로 분리

---

## 6. UI/UX 원칙

* 모바일 최적화 (최우선)
* 3초 내 게시글 등록 가능
* 복잡한 입력 금지
* 일본 사용자 기준 UI

---

## 7. 개발 규칙

* 항상 “완성형 코드” 한번에 생성
* 파일 나누지 말고 전체 구조 출력
* 실행/테스트/디버깅 금지
* 추론 과정 생략

출력 형식:
[프로젝트 구조]
[파일별 코드]

설명 최소화

---

## 8. 토큰 최적화 규칙

* 한 요청 = 한 기능 완성
* 반복 수정 금지
* 긴 설명 금지
* 핵심 코드만 출력

---

## 9. 확장 고려 (SaaS)

향후 구조:

* 학교 → 지역 → 전국 확장
* 학교별 관리자 기능 추가
* 광고 / 프리미엄 기능 추가

멀티 구조:

* school_id 기반 데이터 분리
* 향후 multi-tenant SaaS 전환 가능하게 설계

---
